## Vision

`from-mm.dev` V2 should evolve from a free subdomain tool into a lightweight developer identity and discovery platform for Myanmar developers.

The strongest direction is not to become a full recruiting platform immediately, but to become the best place for Myanmar developers to:

- claim a personal developer identity
- show they are available for work
- showcase their projects and skills
- be discoverable by recruiters, startups, and collaborators

In short:

**V2 should be "the public profile and availability layer for Myanmar developers."**

---

## What V2 Should Focus On

The current product already does one thing well:

- GitHub login
- subdomain registration
- GitHub Pages and Vercel support
- simple dashboard and docs

That means V2 should build on top of this foundation instead of expanding into too many unrelated features at once.

The best V2 scope is:

1. **Developer Profile**
2. **Available for Work status**
3. **Public Waiting Room / Talent Directory**
4. **Basic portfolio/project metadata**
5. **Minimal analytics (views and clicks)**
6. **Optional Netlify support**

This keeps the product focused and realistic.

---

## Recommended V2 Features

### 1. Developer Profile

Each signed-in user should be able to create a simple public profile.

Recommended fields:

- GitHub username
- avatar
- display name
- one-line bio
- skill tags
- location
- work preference (`remote`, `onsite`, `hybrid`)
- employment preference (`full-time`, `part-time`, `contract`, `freelance`)
- portfolio URL
- LinkedIn or contact link
- available-for-work toggle

Why this matters:

- gives the product meaning beyond DNS
- turns each subdomain owner into a discoverable developer
- creates a reusable identity layer for future features

---

### 2. Myanmar Developer Waiting Room

This should be the core V2 feature.

A public page should list developers who opted in as available for work.

Recommended directory features:

- searchable developer list
- filter by skills
- filter by location
- filter by work preference
- filter by availability
- show lightweight social proof on cards such as profile views or portfolio clicks
- simple profile cards with:
  - avatar
  - name / GitHub username
  - one-line bio
  - top skills
  - profile link
  - portfolio link
  - simple analytics such as profile views or project clicks

Why this should be the centerpiece:

- it creates a clear reason for people to visit `from-mm.dev`
- it helps Myanmar developers get discovered
- it is easier to build than a full recruiting platform
- it gives the product a strong community identity

---

### 3. Portfolio Metadata, Not a Full CMS

V2 should **not** try to become a full portfolio builder or media-heavy CMS.

Instead, let users attach lightweight project metadata to their profile:

- project title
- short description
- tech stack
- GitHub repo link
- live demo link
- featured / pinned flag

This is enough for discoverability without making the system too complex.

Recommendation:

- treat `yourname.from-mm.dev` as the main portfolio/project destination
- use `from-mm.dev` as the discovery and indexing layer
- track only minimal analytics at first, such as profile views, portfolio views, and link clicks

This is much simpler and fits the current product direction.

---

### 4. Minimal Analytics

V2 should include a small analytics layer early, but keep it intentionally lightweight.

Recommended analytics for early release:

- profile views
- portfolio page views
- project link clicks
- waiting room card clicks

Recommended product behavior:

- show simple counts on the developer dashboard
- allow a small analytics hint on waiting room cards, such as views or clicks
- avoid charts, heatmaps, or geographic breakdowns in V2
- keep collection privacy-conscious and focused on useful signals only

This gives developers immediate feedback without turning analytics into a separate product.

---

### 5. Netlify Support

Adding Netlify support is a good product improvement, but it should be treated as a secondary feature, not the main story of V2.

Why it is worth adding:

- expands hosting options
- makes the platform more useful
- fits naturally with the current subdomain product

Why it should not lead V2:

- it improves infrastructure, but does not define the product
- the Waiting Room/profile system is more strategic

Recommendation:

- include Netlify support in V2 if implementation is straightforward
- otherwise ship it as V2.1 after the public profile system

---

## What Should Not Be in V2

These ideas are good, but they should be postponed until the core profile + directory experience has traction.

### Defer to Later

- recruiter accounts
- direct messaging
- resume upload
- company pages
- advanced analytics beyond simple views and clicks
- badge customization system
- geographic interest tracking
- complex media upload for projects

Why defer them:

- they add a lot of complexity
- they require more privacy, moderation, and product decisions
- they are not necessary to prove product value

---

## Recommended Product Positioning

Instead of describing V2 as a large all-in-one platform, position it like this:

> **from-mm.dev helps Myanmar developers build a public identity, show availability, and get discovered.**

This is clearer than trying to be:

- a hosting platform
- a recruiting marketplace
- a portfolio CMS
- an analytics product
- a badge platform

all at once.

---

## Suggested V2 Release Scope

### V2 MVP

Ship these first:

- profile editing
- available-for-work toggle
- public waiting room page
- search and filtering
- public profile page
- basic project metadata
- simple analytics for profile views, portfolio views, and card/link clicks

### V2.1

After MVP is stable:

- Netlify support
- improved sorting/filtering
- featured developers or featured projects
- profile activity timestamps
- refine analytics presentation in the dashboard and cards

### V3

Only after real usage:

- recruiter accounts
- saved searches
- direct messaging
- resume access controls
- analytics dashboard
- badges

---

## Product Recommendation Summary

My recommendation is:

- keep the current subdomain system as the foundation
- make the **Waiting Room** the main V2 feature
- add lightweight public developer profiles
- keep portfolio support simple
- add minimal analytics early, limited to useful views and clicks
- avoid recruiter platform complexity for now
- treat Netlify as a useful expansion, not the identity of V2

If V2 is too broad, it will be hard to ship and hard to explain.

If V2 is focused on **developer identity + availability + discoverability**, it can become a clear and meaningful product for Myanmar developers.
